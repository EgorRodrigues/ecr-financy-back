Previsão Financeira (Financial Forecast)

Esta página explica o arquivo financial_forecast.py de forma detalhada e visual.

Income + Expense Período Status projetado/confirmado

O que este arquivo faz

A função get_financial_forecast(session, start_date, end_date) monta uma lista de itens de previsão (ForecastItem) combinando:

  • Receitas (Income) dentro do período
  • Despesas (Expense) dentro do período
  • Categoria (nome da Category) quando existir

O resultado final é uma linha do tempo (ordenada por mês) de “entradas” e “saídas”, marcando se cada item já aconteceu (confirmado) ou ainda é esperado (projetado).

Mapa visual

  1. Ajusta datas Normaliza start_date e end_date para cobrir “meses financeiros”.
  2. Define mês Converte cada data em uma chave YYYY-MM via get_period_str.
  3. Busca receitas Seleciona incomes pendentes/recebidos no período (data efetiva = receipt/due).
  4. Busca despesas Seleciona expenses pendentes/pagas no período (data efetiva = payment/due).
  5. Enriquece com categoria Outer join com Category; lida com SQLite x Postgres na comparação de UUID.
  6. Normaliza e ordena Mapeia status para projetado/confirmado, remove datas nulas e ordena por mês.

1) Ajuste de período (“mês financeiro”)

A função primeiro “ajusta” start_date e end_date para garantir que o período coberto seja consistente para agregação mensal.

Referência: ajuste de datas. 

flowchart LR
  S["start_date"] --> SA{"start_date.day >= 1?"}
  SA -->|sim| S1["start_date = dia 1 do mês atual"]
  SA -->|não| S2["start_date = dia 1 do mês anterior"]

  E["end_date"] --> EA{"end_date.day >= 26?"}
  EA -->|sim| E1["end_date = dia 1 do próximo mês"]
  EA -->|não| E2["end_date = dia 1 do mês atual"]

2) Como o mês (YYYY-MM) é calculado

O helper get_period_str(d) transforma uma data em uma string no formato YYYY-MM. Ele contém uma regra que “empurra” o mês para frente se day >= start_day (neste arquivo, start_day é 1).

Referência: get_period_str. 

flowchart LR
  D["date d"] --> X{"d.day >= start_day"}
  X -->|sim| M["month += 1 (com ajuste ano)"]
  X -->|não| N["mantém month"]
  M --> O["retorna YYYY-MM"]
  N --> O

3) Consulta de receitas (Income)

A consulta de receitas usa duas “colunas calculadas”:

  • Data efetiva: se a income está recebido e tem receipt_date, usa ela; caso contrário, usa due_date.
  • Valor efetivo: se recebido, usa total_received (ou amount se estiver nulo); caso contrário usa amount.

Referência: bloco Incomes. 

sequenceDiagram
  autonumber
  participant DB as Postgres
  participant Q as Query Incomes
  participant R as results[]

  Q->>DB: SELECT incomes + category (outer join)
  DB-->>Q: rows
  loop cada row
    Q->>Q: date = receipt_date se recebido, senão due_date
    Q->>Q: amount = total_received/amount
    Q->>R: adiciona ForecastItem(type=income, status=confirmado/projetado)
  end

4) Consulta de despesas (Expense)

O bloco de despesas é equivalente ao de receitas, mas com as colunas:

  • Data efetiva: se está pago e tem payment_date, usa ela; caso contrário usa due_date.
  • Valor efetivo: se está pago, usa total_paid (ou amount se estiver nulo); caso contrário usa amount.

Referência: bloco Expenses. 

sequenceDiagram
  autonumber
  participant DB as Postgres
  participant Q as Query Expenses
  participant R as results[]

  Q->>DB: SELECT expenses + category (outer join)
  DB-->>Q: rows
  loop cada row
    Q->>Q: date = payment_date se pago, senão due_date
    Q->>Q: amount = total_paid/amount
    Q->>R: adiciona ForecastItem(type=expense, status=confirmado/projetado)
  end

5) Compatibilidade: SQLite vs Postgres (join por UUID)

O arquivo detecta o dialeto do banco. Se for SQLite, ele faz join usando cast para texto (UUID = TEXT). Se for Postgres, faz join normal (UUID = UUID).

Referência: dialect check + join_cond. 

flowchart LR
  A[session.get_bind().dialect.name] --> B{sqlite?}
  B -->|sim| C[Income.category_id == cast(Category.id, Text)]
  B -->|sim| D[Expense.category_id == cast(Category.id, Text)]
  B -->|não| E[Income.category_id == Category.id]
  B -->|não| F[Expense.category_id == Category.id]

6) Montagem do resultado e ordenação

  • Ignora linhas com date nula.
  • Mapeia status: recebido/pago → confirmado, pendente → projetado.
  • Usa “Sem Categoria” quando não há categoria associada.
  • Ordena pelo campo month (string YYYY-MM).
