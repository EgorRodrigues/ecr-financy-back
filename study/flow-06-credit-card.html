Cartão de Crédito (Fatura + Transações + Parcelamento)

Este módulo amarra: credit_card_transactions → credit_card_invoices → expenses (uma expense por fatura).

Visão geral

Transações
/credit-card-transactions (router)
Faturas
/credit-card-invoices (router)
Relação principal
Cada transação pode ter invoice_id; cada fatura pode ter expense_id.
Sincronização
Alterações em invoice.amount/status/due_date/payment_date sincronizam na expense ligada; mudanças no valor da fatura também atualizam expense.amount.

Pré-requisitos (configuração)

  • A conta deve ser type=credit_card (ver regras do schema de accounts).
  • A conta precisa ter closing_day e due_day para o cálculo do período da fatura (senão a criação automática de fatura falha).
  • Para criar fatura (manual ou automática), a conta precisa ter contact_id vinculado (fornecedor do cartão). Caso contrário, a criação da fatura falha.
  • A conta pode ter category_id (apenas credit card) e isso é reutilizado na expense criada para a fatura.

Referências: schemas/accounts.py, ensure_invoice_for_transaction, create_invoice.

Fluxo A — Compra (1 transação)

Endpoint
POST /credit-card-transactions/ (router)
Regras
Se account_id e due_date existirem, tenta garantir a fatura correspondente e amarrar o invoice_id na transação; se a transação não estiver cancelado, soma o amount na fatura.
Falha na fatura
Erros ao criar/atualizar fatura são capturados: a transação ainda é criada, mas sem invoice_id.

Referência: create_credit_card_transaction.

Diagrama

sequenceDiagram
  autonumber
  actor Client as Cliente
  participant API as Router (/credit-card-transactions)
  participant Dep as get_db
  participant Repo as create_credit_card_transaction
  participant Inv as ensure_invoice_for_transaction
  participant DB as Postgres

  Client->>API: POST /credit-card-transactions/ + payload
  API->>Dep: Depends(get_db)
  API->>Repo: create_credit_card_transaction(session, payload)
  alt payload tem account_id e due_date
    Repo->>Inv: ensure_invoice_for_transaction(account_id, due_date)
    Inv->>DB: (cria ou busca invoice e expense)
    Repo->>DB: update_invoice_amount(+amount) se status != cancelado
  else sem dados de invoice
    Repo-->>Repo: invoice_id permanece null
  end
  Repo->>DB: INSERT credit_card_transactions + COMMIT
  API-->>Client: CreditCardTransactionOut

Fluxo B — Compra parcelada (installments)

Endpoint
POST /credit-card-transactions/installments (router)
Regra
installments_total precisa ser >= 2 (senão 400).
Datas
first_due_date (ou issue_date) + soma mês a mês; se o dia não existir no mês alvo, ajusta para o último dia do mês.
Arredondamento
Divide amount_total por N com ROUND_HALF_UP em 2 casas; a última parcela ajusta o resto para fechar o total.
Impacto na fatura
Cada parcela tenta garantir a fatura do seu due_date e soma o valor da parcela na fatura (se status != cancelado).

Referência: create_installment_transactions.

Diagrama

sequenceDiagram
  autonumber
  actor Client as Cliente
  participant API as Router (/credit-card-transactions)
  participant Dep as get_db
  participant Repo as create_installment_transactions
  participant Inv as ensure_invoice_for_transaction
  participant DB as Postgres

  Client->>API: POST /credit-card-transactions/installments + payload
  API->>Dep: Depends(get_db)
  API->>Repo: create_installment_transactions(session, payload)
  Repo->>DB: INSERT installment_group
  loop parcela 1..N
    Repo->>Repo: calcula due_date e installment_amount
    Repo->>Inv: ensure_invoice_for_transaction(account_id, due_date)
    Inv->>DB: (cria ou busca invoice e expense)
    Repo->>DB: update_invoice_amount(+installment_amount) se status != cancelado
    Repo->>DB: INSERT credit_card_transactions (com invoice_id)
  end
  Repo->>DB: COMMIT
  API-->>Client: group + transactions

Fluxo C — Gestão do parcelamento

  • Listar grupos: GET /credit-card-transactions/installment-groups (agregados e próximo vencimento)
  • Detalhar grupo: GET /credit-card-transactions/installment-groups/{group_id} (parcelas ordenadas)
  • Cancelar grupo: POST /credit-card-transactions/installment-groups/{group_id}/cancel marca cada parcela como cancelado e, se ela tiver invoice_id, desconta o valor da fatura.
  • Desativar grupo: POST /credit-card-transactions/installment-groups/{group_id}/deactivate apenas seta active=false no grupo e nas parcelas (não mexe no valor da fatura).

Referências: list_installment_groups, cancel_installment_group, deactivate_installment_group.

Diagrama (cancelar vs desativar)

sequenceDiagram
  autonumber
  actor Client as Cliente
  participant API as Router (/credit-card-transactions/installment-groups)
  participant Dep as get_db
  participant Repo as (cancel|deactivate)_installment_group
  participant DB as Postgres

  Client->>API: POST /{group_id}/cancel ou /{group_id}/deactivate
  API->>Dep: Depends(get_db)
  API->>Repo: executa ação
  alt cancel
    Repo->>DB: SELECT transactions do grupo
    Repo->>DB: para cada tx != cancelado, update_invoice_amount(-amount) e status=cancelado
  else deactivate
    Repo->>DB: group.active=false e tx.active=false
  end
  Repo->>DB: COMMIT
  API-->>Client: group + transactions

Fluxo D — Fatura (Invoice) e Expense automática

Criação manual
POST /credit-card-invoices/ (router)
Criação automática
Chamando ensure_invoice_for_transaction ao criar transações (inclusive parceladas).
Expense criada junto
Ao criar uma fatura, o sistema cria uma Expense com descrição “Fatura Cartão ...” e guarda o vínculo em CreditCardInvoice.expense_id.
Fornecedor/categoria
A Expense usa contact_id e category_id da conta do cartão. Se a conta não tiver contact_id, a criação da fatura falha.

Referências: create_invoice, update_invoice_amount.

Diagrama (criação de fatura)

Regras de negócio — validações e bloqueios

  • Criação automática de fatura depende de closing_day e due_day na conta; se faltar, ensure_invoice_for_transaction lança erro (e no create da transação esse erro é capturado, criando a transação sem invoice_id).
  • Ao criar fatura, a ausência de account.contact_id impede o processo (gera erro no endpoint de faturas e também pode impedir criação automática via transações).
  • A fatura tem unicidade por período: (account_id, period_start, period_end).
  • Edição/remoção de Expense vinculada a fatura é bloqueada no módulo de expenses (não permite update/delete se existir CreditCardInvoice.expense_id = expense.id).
  • Ao atualizar ou deletar uma transação, o sistema ajusta o valor da fatura pelo delta do amount e/ou mudanças de status (cancelado).

Referências: ensure_invoice_for_transaction (validação), UniqueConstraint da fatura, bloqueio em expenses (update/delete), update_credit_card_transaction (delta da fatura), delete_credit_card_transaction.
