Cadastro de Cliente/Fornecedor (Contacts)

Cadastro de um contato do tipo customer ou supplier.

Resumo

Endpoint
POST /contacts/ (contacts.py)
Persistência
repositories/contacts.py:create_contact
Tabela
contacts (models/contacts.py)
Isolamento por tenant
A sessão já está “apontada” para o schema do tenant por get_db.

Validações

  • person_type normaliza sinônimos: pf/fisica → individual, pj/juridica → company.
  • Se document for informado, valida CPF/CNPJ conforme person_type (ValueError → 422).
  • Não há checagem de duplicidade (ex.: documento/email) no fluxo atual; é um insert direto.

Referência: schemas/contacts.py:ContactCreate.

Diagrama (sequência)

sequenceDiagram
  autonumber
  actor Client as Cliente (Front/CLI)
  participant API as FastAPI Router (/contacts)
  participant Auth as verify_token/decode_token
  participant Dep as dependencies.get_db
  participant Val as Pydantic (ContactCreate)
  participant Repo as repositories.create_contact
  participant DB as Postgres (schema do tenant)

  Client->>API: POST /contacts/ (Bearer JWT) + payload
  API->>Auth: Depends(verify_token)
  Auth-->>API: payload (sub=UUID do usuário)
  API->>Dep: Depends(get_db)
  Dep->>DB: (garante schema + SET search_path)
  API->>Val: Valida/normaliza payload
  alt Payload inválido (ex.: CPF/CNPJ)
    Val-->>API: erro (ValueError)
    API-->>Client: 422 Unprocessable Entity
  else Payload válido
    Val-->>API: payload validado
    API->>Repo: create_contact(session, payload)
    Repo->>DB: INSERT contacts + COMMIT
    Repo-->>API: ContactOut
    API-->>Client: 200 OK + ContactOut
  end
