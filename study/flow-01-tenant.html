Criação de um novo tenant (provisionamento lazy)

O schema do tenant é criado automaticamente no primeiro acesso ao banco por um usuário.

Depends(get_db) tenant_<sub> Alembic upgrade head

Mapa visual

  1. Request autenticado Qualquer endpoint com Depends(get_db).
  2. Resolve tenant Calcula tenant_<sub_sem_hifen> a partir do JWT.
  3. Provisiona schema Se não existir, cria schema e roda migrações Alembic.
  4. Aponta a sessão Executa SET search_path para o schema do tenant.
  5. Fluxo do endpoint CRUD/relatórios seguem usando a mesma sessão.

Resumo

O que é “tenant”
Um schema Postgres por usuário: tenant_<sub_sem_hifen>.
Como é criado
Não existe endpoint explícito de “criar tenant”. A criação ocorre automaticamente no primeiro uso do banco pelo usuário.
Onde ocorre
dependencies.py:get_db +
postgres.py:ensure_tenant_schema

Interações que disparam

Qualquer chamada HTTP autenticada a um endpoint que injeta session: Session = Depends(get_db), desde que o schema do usuário ainda não exista no banco.

  • Cliente faz request com Authorization: Bearer <jwt> contendo sub
  • O handler usa Depends(get_db)
  • Se tenant_<sub_sem_hifen> não existe, o backend cria o schema e roda as migrações

Exemplos de endpoints estão em main.py.

Diagrama (sequência)

sequenceDiagram
  autonumber
  actor Client as Cliente (Front/CLI)
  participant API as FastAPI Router
  participant Auth as verify_token/decode_token
  participant Dep as dependencies.get_db
  participant DB as Postgres
  participant Alembic as Alembic (upgrade head)

  Client->>API: Request HTTP (Bearer JWT)
  API->>Auth: Depends(verify_token)
  Auth-->>API: payload (sub=UUID do usuário)
  API->>Dep: Depends(get_db)
  Dep->>Dep: schema_name = "tenant_" + sub.replace("-", "")
  Dep->>DB: SELECT schema exists?
  alt Schema NÃO existe (novo tenant)
    Dep->>DB: CREATE SCHEMA tenant_...
    Dep->>Alembic: command.upgrade(head) com target_schema=tenant_...
    Alembic->>DB: CREATE SCHEMA IF NOT EXISTS "tenant_..."
    Alembic->>DB: SET search_path TO "tenant_...", public
    Alembic->>DB: Cria/atualiza tabelas + alembic_version no schema
  else Schema já existe
    Dep-->>DB: (nenhuma criação)
  end
  Dep->>DB: SET search_path TO tenant_...
  Dep-->>API: Session (apontando para o schema do tenant)
  API-->>Client: Response (fluxo normal do endpoint)

Referências: dependencies.py, postgres.py, alembic/env.py.

Observações

  • O “tenant id” é derivado do sub do JWT (não existe tabela tenants).
  • O script em migrate_tenants.py só aplica migrações em schemas existentes com prefixo tenant_.
